---

## §1 功能定位

**一句话**：把用户粘贴的账单文本一次性识别成结构化账目，用户预览确认后批量入库。

**核心场景**（何时你会用它）：
- 几天没记账，想补一段时间的流水
- 月末盘点，从支付宝/微信复制一段账单进来做"回溯分析"
- 面试演示：拿一段真实账单展示"AI 处理非结构化文本"的能力

**明确边界（MVP 不做）**：
- ❌ CSV 文件上传 —— 用户门槛高（App 里找导出+邮箱验证）
- ❌ 截图 OCR —— 引入图片理解 API，复杂度和成本大幅上升，V2 再说
- ❌ 智能去重算法 —— 预览列表用户自己能看到重复，取消勾选就好
- ❌ 自动同步支付宝/微信 —— 平台不开放，且违背"数据不出本机"叙事

## §2 用户流程

```
① 点左侧"📋 批量导入"图标
     ↓
② 右侧滑入面板 → 大文本框 + "解析"按钮
     ↓
③ 粘贴任意格式的账单文本，点"解析"
     ↓
④ 前端 loading 2-5 秒 → LLM 独立上下文解析 → 返回结构化列表
     ↓
⑤ 面板上半折叠成"原文（点击展开）"，下半展示账目预览
     ↓
⑥ 用户逐条勾选、修改金额/分类、删除不想要的
     ↓
⑦ 点底部"入账 N 笔"
     ↓
⑧ 前端调 /api/bulk_add → 后端逐条过 Pre Hook → 写入 SQLite
     ↓
⑨ 面板关闭，主对话流插入一条系统消息："✅ 已批量入账 N 笔"
     ↓
⑩ 账本视图刷新，日志抽屉出现 N 条 bulk_add 记录
```

## §4 技术架构

### 数据流

```
[前端 · 批量导入面板]
      ↓ POST /api/parse_bulk  {"text": "..."}
[后端 · 独立解析上下文（迷你 Subagent）]
      ├── 独立 System Prompt（专用解析器身份）
      ├── 注入 categories.md（分类规则）
      ├── 注入今日日期（相对日期换算）
      └── LLM 调用 · JSON 强制输出模式
      ↓ 返回结构化列表 [{amount, category, date, note, confidence, warning}]
[前端 · 预览+编辑]
      ↓ 用户勾选+确认，点"入账 N 笔"
      ↓ POST /api/bulk_add  {"items": [...]}
[后端 · 批量入库]
      ├── 逐条过 Pre Hook 安检
      ├── 逐条走 handle_add（SQLite 写入）
      ├── 逐条写 Post Hook 日志
      └── 收集成功/失败结果
      ↓ 返回 {"success": N, "failed": [...]}
[前端 · 关闭面板 + 系统消息 + 账本刷新]
```

### 新增后端接口

**`POST /api/parse_bulk`** —— 解析
- Input: `{"text": string}`（≤ 5000 字符）
- Output: `{"items": [{"tmp_id": int, "amount": float, "category": str, "date": str, "note": str, "confidence": "high"|"medium"|"low", "warning": str|null, "type": "expense"|"refund"|"transfer"}]}`
- 内部：调 llm.chat() 但**用独立 messages 数组**（不走 agent.SESSION）

**`POST /api/bulk_add`** —— 批量入库
- Input: `{"items": [{amount, category, date, note}]}`（不带 tmp_id/confidence 等前端字段）
- Output: `{"success": int, "failed": [{"index": int, "reason": str}]}`
- 内部：逐条 pre_tool_use → handle_add → post_tool_use，收集结果

### 独立解析 System Prompt（关键设计）

```
你是账单解析器。任务：把用户粘贴的账单文本转换成结构化 JSON 数组。

规则：
1. 只提取真正的**消费支出**记录；退款、收入、转账入账要标记 type 但默认不建议入账
2. 金额只取原文里明确的数字，不许估算或推断
3. 日期换算成 YYYY-MM-DD 格式；如果原文只有"昨天/今天"，按【今天=YYYY-MM-DD】换算
4. 分类必须来自下面的 categories.md，找不到匹配就归"其他-未分类"
5. 备注取原文里的商家名/描述，简短（≤ 15 字）
6. 拿不准的字段（比如分类），confidence 设为 low，warning 说明原因
7. 返回必须是合法 JSON 数组，每项包含 tmp_id, amount, category, date, note, confidence, warning, type

【今天日期】{today}
【分类规则】
{categories_md}
```

### 前后端逐条错误处理

```
Pre Hook 失败 → 该条标记 failed，其他继续
handle_add 抛异常 → 该条标记 failed（记 exception message），其他继续
最终返回 {"success": 8, "failed": [{"index": 3, "reason": "..."}]}
前端展示"✅ 成功 8 条 · ❌ 失败 2 条（点击查看详情）"
```

### 复用现有部件

- ✅ `hooks.pre_tool_use` —— 安检门原样复用
- ✅ `hooks.post_tool_use` —— 每条入库写一行日志（工具名="bulk_add"）
- ✅ `skills.load_skill('categories.md')` —— 解析时注入
- ✅ `db.get_conn()` —— 直接写 SQLite
- ✅ `llm.chat()` —— 复用模型客户端

**不复用**：`agent.run_turn()`——因为批量入库不需要 Agent Loop 的"模型再想一遍"。

## §5 关键设计决策 · 3 条

### 决策 1：解析走独立干净上下文（迷你 Subagent）

**做什么**：解析器不使用聊天 session 的 messages 历史，每次批量解析都从零构造 messages。

**为什么**：Agent Loop 的 session 有污染风险——最近对话里出现过的分类/金额可能干扰当前解析。独立上下文让每次批量都可预测、可测试。

**架构对应**：完整 Harness 图谱里的 **s06 Subagent** —— 用干净上下文隔离特定任务。我没做完整 Subagent 系统，因为只有一个隔离任务；但架构层面的隔离已经存在。

**面试话术**：
> "批量解析走独立上下文，这是 Subagent 思想的最小实现。不做完整 Subagent 系统是场景决定的——只有一个隔离任务时，多加抽象是过度工程。"

### 决策 2：批量必预览 · 跳过 Agent Loop 直接入库

**做什么**：解析后必须人工勾选确认；确认后批量入库不走 `/api/chat`，直接调新接口 `/api/bulk_add`。

**为什么**：
- 批量的错误代价放大 —— 一次错就错一堆 —— 必须人工兜底
- 用户已经在预览里明确了意图，Agent Loop 的"让模型再想一遍"是浪费成本
- Pre Hook 安检门保留 —— 安全兜底不因为跳过 Agent 而消失

**信任分层设计**：
```
单笔口头 → AI 全权决策 → 入库（Pre Hook 兜底）
批量粘贴 → AI 建议 → 用户逐条审 → 批量入库（跳过 Agent Loop、Pre Hook 保留）
删/改 →   AI 建议 → 用户确认卡 → 单条入库（现有权限管线）
```

**面试话术**：
> "AI 自主性的边界由风险决定。用户明确知道自己要做什么的时候，让 AI 靠边——这是一个反直觉的架构选择，但它把成本和风险都做到了最优。"

### 决策 3：MVP 明确不做的三件事

**不做智能去重**：预览列表用户自己看得到重复。工程上简单是最优。

**不做 OCR**：图片理解 API 引入外部依赖 + 成本。粘贴够用了。

**不做 CSV 上传**：用户门槛高（App 找导出+邮箱验证）；不同平台格式不一，兼容性成本大。

**面试话术**：
> "不做的三件事都是'看起来必要但没有数据支撑'的功能。MVP 应验证的是核心假设——'粘贴 → 解析 → 批量入库'这个链路能不能跑通、准确率够不够。等我拿到真实使用数据，再谈要不要补 OCR/去重/CSV。"

## §6 边缘情况处理规则表

| 情况 | 处理 |
|---|---|
| 用户粘贴内容里有"退款"关键词 | AI 标记 `type: refund`，**默认不勾选** |
| 用户粘贴内容里有"充值/转入" | 标记 `type: income`，**默认不勾选** |
| 转账给个人（借钱/AA垫付） | 标记 `type: transfer`，归入"其他-人情往来"，**默认勾选**（属支出） |
| 某条解析后 confidence=low | 前端行首显示 ⚠️ 图标 + warning 悬浮说明 |
| 单笔金额 > 100 万 | 高亮红色 + 强制用户点确认才能勾选（防解析错位小数点） |
| Pre Hook 校验失败（比如分类不在集合里） | 该条标记 failed，展示原因，其他条不阻塞 |
| 粘贴文本 > 5000 字符 | 前端拦截：提示"请分批粘贴，每次最多约 100 笔" |
| LLM 返回不是合法 JSON | 后端重试 1 次；仍失败返回错误"解析失败，请调整粘贴内容格式" |
| 全部条目均失败 | 前端不关闭面板，展示错误详情，允许用户直接编辑重试 |
| 用户勾选 0 条点入账 | 按钮禁用，不允许提交空 |

## §7 面试叙事点

这个功能新增了三个可讲的技术亮点，值得写进 [demo_script.md](demo_script.md) 的追问预案：

**亮点 1：迷你 Subagent 实践**
- 现场演示：主对话记账 → 打开批量导入 → 讲"这是一个独立干净上下文"
- 面试价值：证明你不只做了 Agent Loop，也理解为什么有时候要**不用** Agent Loop

**亮点 2：信任分层的安全设计**
- 场景描述：单笔零确认 / 批量必预览 / 删改必确认
- 面试价值：证明你把安全设计做成了**跟场景匹配的分层策略**，不是一刀切

**亮点 3：结构化输出 + JSON 强制模式**
- 技术描述：用 LLM 的 JSON mode，避免自由文本输出的解析噪声
- 面试价值：证明你懂"LLM 的输出可靠性"是产品经理要主动干预的技术选择

1. 后端 `/api/parse_bulk` 接口 + 独立 System Prompt + JSON schema 校验（1.5 小时）
2. 后端 `/api/bulk_add` 接口 + 逐条 Pre Hook + 错误收集（1 小时）
3. 前端批量导入抽屉 UI + 文本框 + 解析按钮（1 小时）
4. 前端预览列表 + 勾选/编辑/删除（1 小时）
5. 集成 + 真实账单测试 + demo_script.md 更新（0.5-1 小时）
